§19 feat(dma): DMA Problem A (mem-to-mem) via HAL — DMA1_CH1 RAM->RAM PASS on board + formal report + KB KI-ENV-0007

First real DMA of Buoi_12: rung 0_1 (HAL) of Problem A (mem-to-mem). The CPU only
declares the "conveyor belt" then steps aside — no polling, no interrupt — and the
result is verified by the register fingerprint read over C-SPY. This is cell (star) =
(Problem A, HAL rung) on the new roadmap map.

Code / build:
- Manufacturer_Package/No.0_...Draft/.../1_Application/Buoi_12/main_buoi12_dma.c
  A single main.c; the API layer is picked by an #ifdef STAGE_0_x label (the label
  IS a macro name — NOT `#if STAGE == 0_1`, which crashed with Pe014; see KB
  KI-ENV-0007). Stage01_MemToMem() sets up DMA1_Channel1 via
  HAL_DMA_Init()+HAL_DMA_Start() to copy 8 words RAM->RAM.
- test.ewp: config `DMA_B12_S01_HAL` (base=Debug) — add USE_HAL_DRIVER, keep
  STM32F103xB, drop USE_FULL_LL_DRIVER. Registers main_buoi12_dma.c + a Vendor_HAL
  group (hal_dma.c); cross-`<excluded>` so the LL `Debug` build is untouched.
  Minimal HAL set: only hal_dma.c (no hal.c/system_stm32f1xx.c — startup never calls
  SystemInit, the linker dead-strips the rest).
- git mv the vendor set into the Draft: hal_dma(.c/.h/_ex.h), hal_cortex(.c/.h),
  hal_conf(.h) (tailored: only the extracted modules stay enabled).
- test.ewd (C-SPY block of the new config, created via GUI) is NOT committed —
  per KI-IAR-0006 only .ewp is committed; .ewd stays as a working-tree "M".

Build `iarbuild -build DMA_B12_S01_HAL`: 0 errors, 0 warnings, Linking OK.
Run (ST-Link V2, pure SWD, C-SPY) — PASS: CNDTR1=0, ISR.TCIF1=1, CCR1=0x4AC1
(MEM2MEM|word|MINC|PINC|EN), CPAR1=0x20000000, CMAR1=0x20000064, g_dst == g_src
(0x11111111 ... 0x88888888). Real evidence under Buoi_12/logs/ + Buoi_12/assets/.
